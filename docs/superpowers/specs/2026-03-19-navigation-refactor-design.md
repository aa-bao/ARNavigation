# 导航定位重构设计

- 日期: 2026-03-19
- 主题: 医院室内导航系统重构，聚焦扫码校准、单段路径规划、指南针模式与 AR 模式统一
- 状态: 已获用户确认，可进入实现计划阶段

## 1. 背景与目标

当前系统的导航能力存在三类职责混杂的问题：

- 后端以完整路径节点和文字说明为中心，服务接口偏“后台管理式路径规划”
- 小程序指南针页自行维护路径推进、距离计算和方向提示
- 小程序 AR 页又叠加了蜂鸟定位与独立的箭头逻辑

这导致导航链路不清晰：

- 位置真值来源不统一
- 两种导航模式共享能力不足
- 前端页面重复持有路径与导航状态
- A* 服务在搜索过程中频繁查库，扩展能力有限

本次重构将系统收敛为如下模型：

- 用户通过二维码扫描校准当前位置
- 后端只返回“当前扫码点到下一个扫码点”的单段路径
- 前端维护统一的导航会话
- 指南针模式与 AR 模式共享同一段路径数据
- 删除蜂鸟定位等连续定位链路

## 2. 设计原则

### 2.1 位置真值唯一

系统中唯一可信的位置来源是二维码扫描结果。陀螺仪和设备姿态仅用于朝向同步，不参与位置推算。

AR 页面中的三维内容以最近一次扫码节点为局部锚点。用户离开该锚点后，系统只继续提供朝向式引导，不宣称叠加内容仍与真实世界保持绝对空间对齐。

### 2.2 单段导航优先

导航不再一次性消费整条终点路径，而是按“当前扫码点 -> 下一个扫码点”分段返回。用户到达下一校准点后重新扫码，再继续下一段。

### 2.3 双模式共用一套会话

指南针模式与 AR 模式共用一份导航会话，不允许两页各自维护独立路径请求、路径解析和步进逻辑。

### 2.4 坐标语义统一

全链路统一采用“米”为单位。

- 数据库 `(x, y)` -> Three.js `(X, Z)`
- 数据库 `floor` -> Three.js `Y`
- `Y = floor * 4.5`

后端输出与前端渲染必须基于同一坐标规则，避免代价函数、可视化和朝向计算使用不同语义。

节点查询接口与单段导航接口都应返回同一坐标语义：

- `planarX`: 数据库 `x`
- `planarY`: 数据库 `y`
- `worldX`: Three.js `X`
- `worldY`: Three.js `Y = floor * 4.5`
- `worldZ`: Three.js `Z = 数据库 y`

前端 session 只保存一种规范化节点模型，不允许页面各自再做轴向猜测。

## 3. 范围

### 3.1 本次包含

- 重构导航后端接口和路径规划服务
- 新增单段导航响应模型
- 重构前端导航会话层
- 重构指南针导航页
- 重构 AR 导航页
- 接入 `threejs-miniprogram`
- 删除蜂鸟定位相关逻辑

### 3.2 本次不包含

- 连续室内定位
- 蓝牙、UWB、视觉 SLAM 等位置估计能力
- 自动判断“用户真实距离下一节点小于 1.5 米”
- 后台维护“关键扫码点”配置能力
- 任何基于位置连续更新的 AR 平移重锚定

## 4. 目标用户流程

### 4.1 导航开始

1. 用户在小程序选择目的地
2. 用户扫描当前位置二维码
3. 前端调用节点查询接口校验二维码
4. 前端调用单段导航接口，请求从当前扫码点到目标点的当前段路径
5. 前端建立导航会话并进入指南针或 AR 模式

### 4.2 单段导航进行中

1. 页面读取统一会话中的单段路径
2. 指南针模式显示当前应朝方向与下一扫码点
3. AR 模式显示相机叠加、路径曲线与前方箭头
4. 页面提示用户前往当前段终点二维码处进行校准

### 4.3 段结束

1. 前端提示“请扫描附近二维码以校准位置”
2. 用户扫描新二维码
3. 系统重新请求下一段路径
4. 若当前扫码点即最终目标，导航结束

## 5. 核心架构

### 5.1 后端架构

建议拆分为以下模块：

- `NavigationController`
- `NavigationSegmentService`
- `HospitalGraphRepository`
- `AStarRoutePlanner`
- `SegmentExtractor`
- `NavigationPointAssembler`

职责划分如下：

#### `NavigationController`

- 提供二维码节点查询接口
- 提供单段导航接口
- 不承载路径算法细节

#### `HospitalGraphRepository`

- 从 `hospital_nodes`、`hospital_edges` 加载图数据
- 将图组织为内存结构，供路径规划器使用
- 避免 A* 搜索过程中逐邻居查库

#### `AStarRoutePlanner`

- 输入起点与终点节点
- 基于内存图执行 A*
- 返回完整路径节点列表

#### `SegmentExtractor`

- 从完整路径中截取“当前扫码点到下一个扫码点”的路径片段
- v1 规则基于“所有节点都可扫码”
- 因此当前段终点等于完整路径中的下一个节点
- 该模块依赖数据不变量：所有参与导航的节点必须具备唯一且非空的 `nodeCode`

#### `NavigationPointAssembler`

- 把路径节点转换成前端消费的坐标点
- 输出三维坐标和必要元数据
- 统一坐标转换

### 5.2 前端架构

建议拆分为以下模块：

- `services/navigation-session.js`
- `services/navigation-api.js`
- `utils/navigation-transform.js`
- `pages/navigation/` 指南针模式页
- `pages/ar/` AR 模式页
- `renderers/ar-scene/` Three.js 渲染模块

职责划分如下：

#### `navigation-session`

- 保存当前导航目标
- 保存当前扫码节点
- 保存当前段终点节点
- 保存当前段路径点
- 保存当前模式
- 管理切模式与重新扫码后的会话刷新

#### `navigation-api`

- 封装二维码节点查询接口
- 封装单段导航接口

#### `navigation-transform`

- 将后端返回的 `points` 转换为二维平面导航结构
- 将后端返回的 `points` 转换为 Three.js `Vector3` 数据
- 集中管理坐标轴映射

#### 指南针页

- 只负责朝向提示与文本提示
- 不自行请求整条路径
- 不自行维护导航步进算法

#### AR 页

- 只负责相机叠加、Three.js 场景和箭头表现
- 不持有独立导航会话

## 6. 数据模型设计

### 6.1 输入模型

单段导航接口请求体：

```json
{
  "startCode": "QR_A_001",
  "targetId": 1024
}
```

字段说明：

- `startCode`: 当前扫码得到的节点编码
- `targetId`: 最终目的地节点 ID

### 6.2 节点查询响应

建议将当前 `/api/navigation/node/code/{nodeCode}` 从“返回列表”改为“返回单个节点”。

示例：

```json
{
  "id": 12,
  "nodeCode": "QR_A_001",
  "nodeName": "门诊大厅入口",
  "floor": 1,
  "planarX": 12.5,
  "planarY": 8.4,
  "worldX": 12.5,
  "worldY": 4.5,
  "worldZ": 8.4,
  "nodeType": "ENTRANCE"
}
```

说明：

- 节点查询接口不再返回模糊的 `x/y`
- 指南针模式使用 `planarX/planarY`
- AR 模式使用 `worldX/worldY/worldZ`

### 6.3 单段导航响应

```json
{
  "segmentStart": {
    "nodeId": 12,
    "nodeCode": "QR_A_001",
    "nodeName": "门诊大厅入口",
    "floor": 1
  },
  "segmentEnd": {
    "nodeId": 18,
    "nodeCode": "QR_A_002",
    "nodeName": "门诊大厅中庭",
    "floor": 1
  },
  "finalTargetId": 1024,
  "isFinalSegment": false,
  "points": [
    {
      "nodeId": 12,
      "nodeCode": "QR_A_001",
      "floor": 1,
      "nodeType": "ENTRANCE",
      "x": 12.5,
      "y": 4.5,
      "z": 8.4
    },
    {
      "nodeId": 18,
      "nodeCode": "QR_A_002",
      "floor": 1,
      "nodeType": "NORMAL",
      "x": 18.1,
      "y": 4.5,
      "z": 15.0
    }
  ]
}
```

说明：

- `points` 为当前段路径点数组
- `x/y/z` 为前端直接消费的 Three.js 坐标
- 该响应不再附带大段文字 directions
- `segmentStart` 和 `segmentEnd` 也应包含与节点查询接口一致的规范化坐标字段，避免 session 在起点与路径点之间做二次转换

## 7. A* 路径规划设计

### 7.1 图模型

节点来源：`hospital_nodes`

边来源：`hospital_edges`

图中每个节点至少包含：

- `id`
- `nodeCode`
- `floor`
- `xCoordinate`
- `yCoordinate`
- `nodeType`

并满足以下数据不变量：

- `nodeCode` 非空
- `nodeCode` 全局唯一
- 所有用于导航的节点都可由二维码扫描得到

每条边至少包含：

- `fromNodeId`
- `toNodeId`
- `distance`
- `isAccessible`
- `directionAngle`

### 7.2 代价函数

基础代价：

`cost = edge.distance`

电梯等待权重：

若本次扩展进入的目标节点 `nodeType == ELEVATOR`，则：

`cost += 15`

该策略符合需求中的“为 ELEVATOR 类型节点增加 15 单位等待权重”。

### 7.3 启发函数

采用 3D 欧氏距离：

- `dx = x1 - x2`
- `dy = (floor1 - floor2) * 4.5`
- `dz = y1 - y2`

`h = sqrt(dx^2 + dy^2 + dz^2)`

说明：

- 不再使用独立的楼层惩罚常量
- 直接与前端 3D 坐标保持一致

### 7.4 路径截取规则

完整路径设为：

`[N0, N1, N2, ..., Nk]`

其中：

- `N0` 为当前扫码点
- `Nk` 为最终目标点

v1 规则：

- 若路径长度为 `1`，则当前点即目标点
- 若路径长度大于 `1`，则单段路径截取为 `[N0, N1]`
- 当前段终点为 `N1`

该规则建立在“所有节点都具有二维码，且都可作为校准点”的前提上。

注意：

该规则最简单，但用户扫码频率高。后续若要减少扫码频率，可再扩展“校准点筛选规则”，但不影响本次架构。

## 8. 前端双模式设计

### 8.1 共享导航会话

建议统一维护如下会话结构：

```json
{
  "targetId": 1024,
  "targetName": "内科诊室 3",
  "currentScannedNode": {},
  "segmentEndNode": {},
  "isFinalSegment": false,
  "segmentPoints": [],
  "currentMode": "compass"
}
```

说明：

- `segmentPoints` 是会话中的唯一规范路径数据
- `polyline2d` 与 `polyline3d` 作为派生数据按需即时计算，不持久化到 session

### 8.2 指南针模式

指南针模式职责：

- 使用统一导航会话中的二维路径
- 监听设备朝向
- 计算当前段方向与设备朝向夹角
- 输出方向提示，如“向左转”“向右前方”“直行”
- 展示当前扫码点和下一扫码点
- 提供重新扫码入口

不再承担以下职责：

- GPS 更新
- 室内连续定位
- 自动推进步数
- 根据假定距离自动判断到达

### 8.3 AR 模式

AR 模式职责：

- 使用统一导航会话中的三维路径
- 使用小程序相机画面作为背景
- 在上层 `webgl canvas` 中渲染路径和箭头
- 利用设备姿态更新相机旋转
- 将最近一次扫码点视为局部锚点，显示“从该锚点出发的朝向式引导”

不再承担以下职责：

- 独立维护路径请求
- 独立维护当前位置
- 接入蜂鸟定位
- 声称三维叠加物在用户移动后仍与真实世界绝对对齐

### 8.4 模式切换

切换规则：

- 指南针模式 -> AR 模式：复用当前会话，不重新请求后端
- AR 模式 -> 指南针模式：复用当前会话，不重新请求后端
- 用户重新扫码后：更新会话并重新请求下一段

## 9. AR 相机与 Three.js 设计

### 9.1 页面结构

AR 页面建议采用以下结构：

- 相机组件显示真实世界画面
- `<canvas type="webgl">` 叠加 Three.js 渲染

Three.js 负责三维引导元素，不负责采集摄像头图像。

### 9.2 相机位置

`THREE.PerspectiveCamera` 初始位置设置为当前扫码节点坐标：

- `camera.position.x = point.x`
- `camera.position.y = point.y`
- `camera.position.z = point.z`

这里的“位置”表示虚拟场景中的用户起点。

用户行走后，不更新相机平移。AR 场景保持以最近一次扫码点为局部参考，仅通过旋转提供方向引导。

### 9.3 相机旋转

通过 `wx.onDeviceMotionChange` 持续更新相机旋转，使手机转动时 Three.js 视角同步变化。

本次设计中，设备姿态用于：

- 同步 AR 视觉方向
- 为指南针模式计算相对朝向

设备姿态不用于：

- 推算用户已前进多远
- 自动确认到达节点

### 9.4 路径绘制

路径点通过 `THREE.CatmullRomCurve3` 生成平滑曲线。

建议渲染要素：

- 发光路径线
- 路径采样点
- 当前段终点高亮

### 9.5 箭头引导

箭头不固定在最终目的地，也不固定在用户正前方屏幕中间。

建议逻辑：

- 基于曲线长度计算一个前方偏移距离，建议约 2 米
- 取曲线对应采样点作为箭头位置
- 取该位置的曲线切线作为箭头朝向
- 使用 `THREE.Mesh` 创建发光箭头

这样当用户转动设备时，箭头仍然稳定地悬挂在前进方向上，更符合“指向目的地”的预期体验。

这里的“稳定”指相对于最近一次扫码建立的局部场景稳定，而不是对真实物理空间的持续绝对锚定。

## 10. 到点与校准策略

### 10.1 本次策略

由于系统不做连续定位，因此不应声称系统可以可靠得知“距离下一节点 < 1.5 米”。

本次采用保守策略：

- 前端始终提供显式“重新扫码校准”入口
- 当当前段终点节点已知时，页面持续展示“请前往下一二维码点并扫描校准”
- 当用户点击“我已到达下一点”或主动发起扫描时，进入下一次扫码流程
- 文案建议为：“请寻找附近二维码并扫描以校准位置”

该规则是前后端共享的确定性触发机制，不依赖系统估算用户与段终点的物理距离。

### 10.2 可选增强

若后续希望弱化扫码频率，可扩展：

- 手动“我已到达下一点”按钮
- 基于相机识别二维码后的自动跳段
- 引入新的连续定位能力

这些都不属于本次实现范围。

## 12. 测试策略

### 12.1 后端测试

应新增或调整以下测试：

- `startCode` 不存在时返回 404/业务错误
- `targetId` 不存在时返回业务错误
- 图数据中存在空 `nodeCode` 或重复 `nodeCode` 时拒绝进入单段导航
- A* 在同层路径上的最短路径正确
- 含电梯节点时权重正确生效
- 起点即终点时返回最终段
- 单段截取结果只包含当前扫码点到下一扫码点
- 返回坐标满足 `y = floor * 4.5`

### 12.2 前端测试

重点验证：

- 扫码后成功建立会话
- 指南针模式正确展示下一扫码点方向
- AR 模式正确渲染曲线与箭头
- 模式切换不丢失当前段状态
- 重新扫码后正确刷新为下一段
- 删除蜂鸟定位后页面不再依赖连续位置更新

### 12.3 手工联调场景

- 同楼层短路径
- 跨楼层路径，经过电梯
- 用户在中途切换指南针和 AR 模式
- 用户扫到错误二维码
- 用户已在目标点扫码

## 13. 风险与约束

### 13.1 姿态漂移

不同机型的设备姿态精度不同，AR 页面可能出现朝向漂移。该问题影响体验，但不影响系统位置真值。

### 13.2 扫码频率高

由于当前假设“所有节点都可扫码且都作为下一校准点”，用户可能需要频繁扫码。该问题是有意识接受的 v1 简化。

### 13.3 坐标系统一要求严格

若前后端任何一侧误将数据库 `y` 当作 Three.js `Y` 使用，AR 路径会出现明显错位，因此坐标转换必须集中实现，避免页面内重复编写。

## 14. 实施建议

建议优先级如下：

1. 先完成后端单段导航接口与统一响应
2. 再抽出前端共享 session 和数据转换层
3. 接着重构指南针模式
4. 最后重构 AR 模式并接入 Three.js
5. 完成后删除蜂鸟定位相关代码

该顺序可以降低联调期间的变量数量，便于快速验证后端单段路径模型是否正确。

## 15. 结论

本次重构的本质不是“换一个 AR 页面”，而是重新定义导航系统的定位真值与状态边界：

- 扫码负责位置
- 姿态负责方向
- 后端负责单段路径
- 前端负责双模式呈现

在此模型下，指南针模式和 AR 模式将共享同一导航会话与同一单段路径数据，系统复杂度显著下降，且后续可以在不破坏主架构的前提下扩展更复杂的校准点策略或连续定位能力。
