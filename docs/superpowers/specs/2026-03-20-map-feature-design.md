# 地图功能设计

- 日期: 2026-03-20
- 主题: 为医院 AR 导航系统新增地图能力，支持小程序地图 tab、导航页地图入口、扫码定位后的地图实时标注，以及 Vue 管理端地图预览与校验
- 状态: 已获用户确认，待用户审阅文档后进入实现计划阶段

## 1. 背景与目标

当前系统已经具备以下基础能力：

- 数据库中存在规范化的医院节点与边数据
- 节点具有 `floor`、`x_coordinate`、`y_coordinate` 坐标
- 小程序已支持二维码扫码定位
- 小程序首页与导航页已维护当前位置、目标点与导航 session
- 用户已产出 1F、2F、3F 平面图，以及一张 3D 总览图

本次新增“地图”功能的目标是：

- 让用户在小程序中查看医院楼层地图
- 用户扫码定位后，在地图上看到自己的当前位置
- 用户选定目的地后，在地图上看到目的地位置
- 用户导航时可从导航页进入地图，边导航边看图
- 管理端提供地图预览能力，辅助验证坐标与图片的一致性

本次设计默认采用用户已确认的前提：

- 地图图片与数据库坐标系一致，可直接复用数据库坐标进行投影

## 2. 设计原则

### 2.1 数据坐标单一来源

地图上的所有位置点、路径点、目的地点，都以数据库节点坐标为唯一来源：

- `floor`
- `x_coordinate`
- `y_coordinate`

前端不维护独立的像素坐标真值，不建立第二套地图坐标体系。

### 2.2 图片只作为底图

PNG 图片仅作为视觉底图，不参与定位计算。实际定位依据仍是：

- 用户扫码得到 `nodeCode`
- 前端通过节点接口获取该节点的标准化坐标
- 前端将坐标映射到地图图片上的像素位置

### 2.3 小程序与管理端共享同一映射规则

小程序端与 Vue 管理端必须使用同一套地图配置和同一套投影规则，避免一端显示正确、另一端偏移。

### 2.4 先做高价值可落地能力

v1 优先提供：

- 地图 tab
- 导航页地图入口
- 当前位置标注
- 目的地标注
- 管理端楼层地图预览

整条路径绘制、拖拽缩放、点选详情等能力作为增量扩展，不阻塞第一阶段上线。

## 4. 已验证前提

### 4.1 数据库坐标具备二维平面语义

根据 [hospital_ar_db.sql.md](/f:/Coding/JavaProject/ARNavigation/sql/hospital_ar_db.sql.md)，`hospital_nodes` 已提供：

- `floor`
- `x_coordinate`
- `y_coordinate`

且坐标分布呈规则平面结构，可直接表达楼层内布局。

### 4.2 绘图脚本直接使用数据库坐标

根据 [plan.ipynb](/f:/Coding/JavaProject/ARNavigation/jupyter/plan.ipynb)：

- 平面图边线使用 `ax.plot([n1[3], n2[3]], [n1[4], n2[4]], ...)`
- 节点使用 `ax.scatter(x, y, ...)`
- 名称使用 `ax.text(x, y + 1.8, ...)`
- 视图范围固定为 `xlim(-8, 48)`、`ylim(-20, 35)`
- 使用 `ax.set_aspect('equal')`

说明当前 PNG 实际上是数据库坐标系的静态导出图，而不是经过二次投影或自由排版的示意图。

### 4.3 唯一需要处理的坐标差异

前端屏幕坐标中：

- `left` 向右增大
- `top` 向下增大

数据库与 matplotlib 坐标中：

- `x` 向右增大
- `y` 向上增大

因此前端渲染时需要做一次 Y 轴翻转。这是实现细节，不影响方案成立。

## 5. 用户流程

### 5.1 普通查看地图

1. 用户点击底部 `地图` tab
2. 系统默认显示当前楼层地图；若无当前位置则默认显示 1F
3. 用户可以切换 1F、2F、3F、3D 总览
4. 若系统已有当前位置，则在对应楼层图上显示当前位置点

### 5.2 扫码后查看当前位置

1. 用户在首页扫码定位
2. 首页将当前位置写入全局 `currentLocation`
3. 用户进入地图页
4. 地图页读取当前位置并在对应楼层图上标注当前位置

### 5.3 选择目的地后查看地图

1. 用户在首页选择目的地
2. 首页将目标点写入全局 `destination`
3. 用户进入地图页
4. 地图页在目标楼层图上显示目的地点
5. 如果当前楼层与目标楼层不同，页面提示需要换层

### 5.4 导航中查看地图

1. 用户已建立导航 session
2. 在 `navigation` 页面点击“查看地图”
3. 系统跳转地图页并以导航视图打开
4. 地图页读取 `currentScannedNode` 与 `destination`
5. v1 当前楼层图显示当前位置；若目标不在本层则显示换层提示
6. 第二阶段再接入 `segmentEndNode`，用于显示当前导航段终点

### 5.5 管理端预览

1. 管理员进入地图预览页
2. 选择楼层或 3D 总览
3. 系统叠加节点点位
4. 管理员按节点类型、名称筛选并查看节点详情
5. 如有需要，可叠加边线校验路径网络是否与图一致

### 6.2 前端职责

前端负责：

### 6.3 共享能力抽取

## 7. 数据与配置设计

### 7.1 地图资源配置模型

建议引入统一配置结构：

```json
[
  {
    "floor": 1,
    "type": "FLOOR_PLAN",
    "name": "1F 平面图",
    "imageUrl": "/static/maps/Hospital_Layout_Floor_1.png",
    "imageWidth": 188145,
    "imageHeight": 188145,
    "minX": -8,
    "maxX": 48,
    "minY": -20,
    "maxY": 35
  }
]
```

说明：

- `minX/maxX/minY/maxY` 来自绘图脚本的坐标显示范围
- `imageWidth/imageHeight` 可选。如果前端按容器百分比定位，则不强依赖原始像素值
- `type=OVERVIEW_3D` 用于 3D 总览图

### 7.2 推荐后端响应模型

建议新增接口：

`GET /api/maps`

返回地图资源配置数组，供小程序与管理端共用。

如需支持管理端边线叠加，建议同时明确以下边界之一：

- 方案 A：复用已有地点与路径关系接口，由管理端自行组织边线
- 方案 B：新增 `GET /api/maps/graph?floor=1`，返回当前楼层节点与边线关系

v1 不要求小程序消费边线接口；边线数据仅为管理端预览增强能力服务。

如暂时不做后端接口，也可以先以前端本地常量实现，但应保持：

- 配置集中
- 两端复用
- 后续易于迁移到后端

### 7.3 节点模型要求

所有参与地图渲染的节点至少需要：

- `nodeId`
- `nodeCode`
- `nodeName`
- `floor`
- `planarX`
- `planarY`
- `nodeType`
- `description`

说明：

- 小程序 `currentLocation` 目前已经在 `app.js` 中保存了 `coordinates.x/y`
- 导航 session 目前已保存 `currentScannedNode`，并已具备后续接入 `segmentEndNode` 的结构基础
- 因此地图页可以直接复用现有全局状态和 session 状态

## 8. 坐标投影设计

### 8.1 归一化公式

对任意节点：

- `normalizedX = (x - minX) / (maxX - minX)`
- `normalizedY = (y - minY) / (maxY - minY)`

### 8.2 前端渲染公式

由于屏幕坐标需要做 Y 轴翻转：

- `leftPercent = normalizedX * 100`
- `topPercent = (1 - normalizedY) * 100`

### 8.3 边界处理

前端需要对归一化结果做裁剪：

- 小于 `0` 则按 `0`
- 大于 `1` 则按 `1`

避免未来节点数据超出边界时 marker 完全丢失。

### 8.4 抽象接口

建议封装函数：

- `getMapConfigByFloor(floor)`
- `projectPlanarPointToMap({ x, y, floor, config })`
- `buildMapMarker(node, markerType)`

这样小程序与 Vue 管理端都不在页面里手写映射公式。

## 11. 降级策略

### 11.1 v1 主方案

按用户确认的 A 方案落地：

- 静态底图
- 坐标投影
- 当前位置、目的地实时标注

对于 `3D 总览图`，v1 明确采用“仅总览展示，不做精确 marker 定位”的规则：

- 小程序端：显示 3D 总览图本身，并在信息卡中展示“当前位置楼层”和“目的地楼层”
- Vue 管理端：允许显示楼层级统计信息或节点列表摘要，但不要求在 3D 图上做精确点位叠加
- 精确点位标注只在 1F/2F/3F 平面图中进行

### 11.2 轻度降级

若个别楼层出现展示偏差，可临时降级为：

- 仅显示当前位置与目的地
- 暂不开放当前导航段终点高亮

若出现以下异常，统一回退为“仅展示底图 + 文案提示”：

- 楼层缺少地图配置
- 图片资源缺失
- 图片加载失败
- 节点楼层值不在已配置楼层集合内

### 11.3 重度降级

若图片资源或投影存在未预期问题，可临时退回：

- 仅显示地图底图
- 不叠加 marker

但从当前脚本与数据核验结果看，预计不需要退到该级别。

## 13. 并发实施策略

用户已明确要求实施规划阶段使用多代理并发执行以提升效率。

因此实施计划应拆成可并行的独立子任务，至少包含：

- 后端地图资源接口或配置落地
- 小程序地图页与 tab 接入
- 小程序导航页地图入口与 session 联动
- Vue 管理端地图预览页
- 共享地图映射工具层

并发约束：

- 不同子任务的写入文件范围需尽量分离
- 公共数据结构与配置文件需先定义接口边界
- 子代理不得回滚其他代理或用户现有改动

推荐并发切分：

1. 子代理 A：小程序地图页、tabBar、导航入口
2. 子代理 B：Vue 管理端地图预览页
3. 主线程：共享地图配置、坐标投影逻辑、必要接口与最终集成

若后端接口需要调整，可再增加一个专门处理后端地图配置输出的子代理。

## 15. 风险与约束

### 15.1 图片裁剪范围依赖脚本导出规则

当前映射默认依赖绘图脚本的：

- `xlim(-8, 48)`
- `ylim(-20, 35)`

如果未来重新导出图片且修改了显示范围，地图配置必须同步更新。

### 15.2 3D 总览图不适合作为精准定位图

`hospital_3d_layout.png` 更适合总览展示，不适合作为单点精准定位载体。因此 v1 中：

- 3D 总览图只做展示
- 实时 marker 精确定位以楼层平面图为主

### 15.3 当前 session 与全局状态存在双来源

小程序当前同时存在：

- `app.globalData.currentLocation`
- `app.globalData.navigationSession`

地图页必须明确优先级，避免当前位置取值不一致。

- 复用现有扫码定位状态
- 将静态地图图片作为统一可视化载体
